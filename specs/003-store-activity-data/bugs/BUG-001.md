# Bug Report: Sync Failures Lack Actionable Diagnostic Reports

**Type**: Spec gap
**Severity**: Medium
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-15
**Status**: Patched
**Patched**: 2026-05-15

## Description

Current sync failures surface only coarse failure categories and brief transient text, which is not enough to help a user or engineer troubleshoot sync and protected-storage failures. The reported requirement is to support a structured diagnostic-report workflow: in production, ask whether to generate a redacted report; in development mode, generate it automatically; in both modes, exclude the Ghostfolio security token and tell the user where the report was written.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 2 - Reuse Token-Locked Stored Data, especially scenario 7 where incompatible new data must inform the user and preserve the previously readable snapshot
- **Affected user story**: Story 3 - Preserve Data Quality And Server Boundaries, where failed validation and replacement flows must fail safely and explain the outcome
- **Affected requirements**: `FR-026` (sync outcomes explain the failure and next step without exposing secrets or unprotected activity data), `FR-030` (no transient failure messages or raw unprotected payloads are persisted for later diagnostics or storage), `FR-037` (incompatible new data must inform the user and keep the previous readable snapshot active), `SEC-001` (tokens excluded from logs and output), `SEC-004` (persisted artifacts must never contain tokens, raw payload fragments, or transient sync-failure messages)
- **Gap identified**: The spec requires safe user-facing failure summaries and strict leakage prevention, but it does not define a supported diagnostic-report artifact, its redaction rules, its storage location, prod versus dev behavior, or the UI flow that discloses report generation. That omission leaves the artifact set driving only generic failure text.

### plan.md

- **Affected sections**: `Technical Context` constraints and testing strategy, `Compatibility And Replacement Rules`, and `Verification Additions`
- **Impact**: The plan needs explicit design for diagnostic report generation, redaction policy, production opt-in versus development auto-generation, report storage location and retention, and automated verification that generated reports exclude tokens, JWTs, raw payload fragments, and any disallowed financial-value detail.

### tasks.md

- **Affected tasks**: `T013` (sync workflow contract coverage for result wording), `T024` (storage-focused result screen wording), `T032` (failure and success result handling), `T044` (quickstart sync-result documentation), `T049` (persisted-artifact security coverage)
- **False completions**: none. The completed tasks match the current artifact set, which only specifies coarse failure categories and brief result text.
- **Missing tasks**: add contract and integration coverage for diagnostic-report prompts and generated-report notices; add runtime and UI work for redacted sync-failure reports; document report location and dev-mode behavior; add security tests proving generated reports never contain tokens, JWTs, raw payload fragments, or unredacted financial values outside the explicitly allowed scope.

## Root Cause Analysis

The current feature artifacts focus on safe, non-secret sync outcomes, but they never define a deeper troubleshooting workflow. That omission propagated into the plan, tests, and implementation, which now expose only generic failure categories and short follow-up text.
