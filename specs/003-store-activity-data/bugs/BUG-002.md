# Bug Report: Ghostfolio Activity Timestamps Break Deterministic Ordering

**Type**: Dependency issue
**Severity**: High
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-15
**Status**: Patched
**Patched**: 2026-05-15

## Description

Investigation showed that Ghostfolio's `/api/v1/activities` `date` field is not reliable as a precise event-time ordering key. Ghostfolio's activity UI captures only a calendar date, so the returned time-of-day can be arbitrary. The current feature artifacts assume the full `occurred_at` timestamp is trustworthy and use only `source_id` as the tie-breaker for same-asset events.

This can make valid same-day activity histories normalize in the wrong order or fail defensibility checks incorrectly. For deterministic same-asset ordering, the application should:

1. ignore the time-of-day portion of `occurred_at` and compare only the calendar date for ordering
2. when the same asset still shares the same normalized `occurred_at`, prefer `BUY` before `SELL`
3. use `source_id` only as the final tie-breaker

The original timestamp should still be stored exactly as received for auditability and report-year derivation. Only the ordering rule is affected.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 3 - Preserve Data Quality And Server Boundaries, especially the deterministic-ordering and non-defensible-history paths plus the edge case for same-asset events that cannot be ordered after normalization
- **Affected requirements**: `Terms Used In This Spec -> Defensible history`, `FR-021`, `FR-022`, `QUAL-001`, `SC-003`
- **Gap identified**: `spec.md` currently defines deterministic same-asset ordering as `occurred_at` followed by `source_id`. That rule assumes the Ghostfolio `date` field carries trustworthy time precision. The spec is missing the required dependency caveat that same-asset ordering must ignore time-of-day precision and then use `activity_type` (`BUY` before `SELL`) before `source_id`.

### plan.md

- **Affected sections**: `Technical Context` constraints, `Constitution Check` external API review, `Full-History Sync Rules`, and `Verification Additions`
- **Impact**: The plan must stop treating the upstream `date` time component as a defensible ordering key. It needs separate rules for storage precision versus ordering precision: preserve the original timestamp for storage and year derivation, but normalize same-asset ordering to date precision, then break ties by `activity_type` and `source_id`. Contract-review notes and verification cases also need fixtures where Ghostfolio returns same-day records with arbitrary time values.

## Root Cause Analysis

Planning and contract review treated Ghostfolio's `date` field as a stable event timestamp with meaningful time precision. That assumption was copied into the spec's defensibility definition, the plan's normalization rules, and the completed ordering tasks and tests. Later investigation showed Ghostfolio's own activity UI does not capture time-of-day, so the dependency behavior does not support the current ordering rule. This is a dependency assumption error that must be corrected in the artifacts before implementation can be considered consistent.
