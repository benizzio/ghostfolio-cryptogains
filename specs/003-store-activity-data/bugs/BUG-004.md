# Bug Report: Optional Ghostfolio Currency Fields Break Sync Validation

**Type**: Dependency issue
**Severity**: High
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-17
**Status**: Open

## Description

Live execution failed with:

`activity "ae8c8ca0-0feb-4303-bc66-c9e7296e02e6" order unit price currency context is incomplete`

The current contract now specifies three separate currency-context tiers with
different optionality rules:

- `Order.currency` is nullable in Ghostfolio. `currency = null` is an allowed
  upstream value and must be preserved as returned.
- `SymbolProfile.currency` is the asset-profile currency tier and is not
  guaranteed by the public `Activity` response contract.
- Authenticated-user `settings.baseCurrency` is the base-currency tier. It is
  optional in the public `UserSettings` contract, but it becomes a current-slice
  contract failure when base-valued activity amounts are present and Ghostfolio
  still does not provide it.

Under the current contract, sync should not fail just because one optional
currency field is absent or null on its own. Currency-context validation should
fail only when all currency fields are absent.

  activity data with explicit per-amount context where Ghostfolio provides it.
- **Affected user story**: Story 3 - Preserve Data Quality And Server
  Boundaries, especially scenario 11, because the current rule rejects missing
  currency metadata even when the source itself emitted that omission.
- **Affected requirements**: `FR-004` (retrieve the Ghostfolio fields needed for
  storage), `FR-018` (preserve each stored monetary amount with its own source
  currency context), `FR-021` (reject missing or contradictory reporting fields
  before persistence), `INT-001` (the slice depends on Ghostfolio exposing the
  needed monetary and currency inputs), `QUAL-001` (test rejection of incomplete
  currency context), and `SC-008` (stored monetary inputs remain tied to the

- **Gap identified**: `spec.md` still describes currency-context rejection in
  broad terms, but it does not carry forward the contract's three-tier
  distinction between optional `Order.currency`, optional
  `SymbolProfile.currency`, and conditionally required authenticated-user
  `settings.baseCurrency`. That leaves room to misread the spec as if any single
  missing currency field is enough to fail sync, instead of applying the
  contract's narrower rule set.

### plan.md

- **Affected sections**: `Technical Context` constraints, `Constitution Check`
  external API review, `Full-History Sync Rules`, and `Verification Additions`.
- **Impact**: The plan still needs the same contract-level distinction. It
  should say that nullable `Order.currency` is preserved as retrieved,
  `SymbolProfile.currency` is not guaranteed per row, and
  authenticated-user `settings.baseCurrency` is also optional. Validation and 
  verification language should then fail only when none of the currency values are present.

### contracts/ghostfolio-sync.md

- **Rechecked status**: The current contract artifact already documents this bug
  more accurately than the original BUG-004 text did.
- **Verified contract rules**:
  - `Activity Field Nullability And Presence Rules` states that `currency = null`
    is allowed and must be preserved, and that `SymbolProfile.currency` is not
    guaranteed by the public response contract.
  - `Authenticated User Contract` states that `settings.baseCurrency` is
    optional in `UserSettings`, but required when base-valued activity amounts
    are present.
  - `Minimum Required Inputs Per Retrieved Activity` states that `currency`

    authenticated-user `settings.baseCurrency` are allowed to be absent,

  - `Normalization And Validation Rules` narrows currency-context failure to all 
    the currency fields not present at the same time.
- **Remaining gap**: `contracts/ghostfolio-sync.md` no longer appears to be the
  primary artifact gap for this bug. The remaining traceability gap is in the
  downstream wording that still overstates how broadly currency-context
  validation should fail.

## Root Cause Analysis

BUG-003 correctly identified that monetary values must not be relabeled under
one record-level currency field, but the follow-up BUG-004 report was written
before the refreshed `003` contract fully documented the current upstream
optionality rules.

The verified contract now distinguishes three separate currency-context tiers:
optional `Order.currency`, optional `SymbolProfile.currency`, and
conditionally required authenticated-user `settings.baseCurrency`. The
remaining issue is that this bug report and some downstream traceability still
describe currency-context validation too broadly, which makes the current `003`
artifact set look less precise than the verified contract actually is.
