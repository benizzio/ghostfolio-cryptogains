# Bug Report: Ghostfolio Nullable Currency Definitions Reject Valid Activity Rows

**Type**: Dependency issue
**Severity**: High
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-18
**Status**: Patched
**Patched**: 2026-05-18

## Description

Production sync data triggered the validation error `activity "ae8c8ca0-0feb-4303-bc66-c9e7296e02e6" order unit price currency context is incomplete`.

The refreshed Ghostfolio contract evidence now shows that the public API exposes looser currency-definition guarantees than the higher-level `003` artifacts still assume:

- `Order.currency` can be `null`
- `SymbolProfile.currency` is optional in the public `Activity` contract
- sync base-currency context comes from authenticated `GET /api/v1/user` data, and `settings.baseCurrency` is optional in the public `UserSettings` contract

`specs/003-store-activity-data/contracts/ghostfolio-sync.md` was tightened to reflect those findings, but `spec.md`, `plan.md`, and `tasks.md` still encode stricter currency-definition assumptions. That mismatch leaves valid production records classified as incomplete currency context during mapping or validation.

A new validation rule should be adopted: sync should fail with a currency-context error only when all three independent currency-definition tiers are not informed for the preserved monetary data. Those tiers are:

- activity `currency` from `Order.currency`
- asset-profile `SymbolProfile.currency`
- authenticated-user `settings.baseCurrency`

Missing one tier by itself must not fail sync when the preserved monetary data still has enough informed currency context through the other independent tier or tiers.

## Root Cause Analysis

BUG-003 corrected the need to preserve per-amount currency identity, but the feature artifacts still treated Ghostfolio currency-definition fields as effectively present whenever related monetary values existed. Production data then exposed valid activity rows where that assumption does not hold.

The contract review was refreshed first and now documents that order currency can be null, symbol-profile currency is not guaranteed by the public contract, and base currency is sourced from authenticated user settings rather than from per-activity payloads. Because the higher-level feature artifacts were not updated in the same pass, implementation and verification work continued against stricter assumptions than the dependency actually guarantees, instead of following the tier-aware rule that only a complete absence of all three independent currency-definition tiers should fail sync.
