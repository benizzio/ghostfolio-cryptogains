# Bug Report: Nullable Ghostfolio Order Currency Breaks Sync Validation

**Type**: Dependency issue
**Severity**: High
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-17
**Status**: Open

## Description

Live execution failed with:

`activity "ae8c8ca0-0feb-4303-bc66-c9e7296e02e6" order unit price currency context is incomplete`

The current `003` artifacts assume that when Ghostfolio returns order-scoped
monetary values such as `unitPrice`, `value`, or `fee`, the matching
`Order.currency` value can be required for current-slice validation.

That assumption is not empirically reliable. In live Ghostfolio data,
`Order.currency` can be `null` while the activity itself is otherwise valid and
worth storing for later slices. The current validation rules therefore reject
upstream data that Ghostfolio can legitimately produce.

This makes the slice impose a traceability requirement that exceeds the current
dependency contract and pulls a later-slice reporting decision into the present
storage slice. The current slice should preserve nullable Ghostfolio financial
fields exactly as provided, and should only require currency-context completion
that Ghostfolio can actually provide itself.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 1 - Sync And Store Full Activity History,
  especially scenario 6, because the slice promises to store mixed-currency
  activity data with explicit per-amount context where Ghostfolio provides it.
- **Affected user story**: Story 3 - Preserve Data Quality And Server
  Boundaries, especially scenario 11, because the current rule rejects missing
  currency metadata even when the source itself emitted that omission.
- **Affected requirements**: `FR-004` (retrieve the Ghostfolio fields needed for
  storage), `FR-018` (preserve each stored monetary amount with its own source
  currency context), `FR-021` (reject missing or contradictory reporting fields
  before persistence), `INT-001` (the slice depends on Ghostfolio exposing the
  needed monetary and currency inputs), `QUAL-001` (test rejection of incomplete
  currency context), and `SC-008` (stored monetary inputs remain tied to the
  correct currency context).
- **Gap identified**: `spec.md` treats missing order-currency context as a
  current-slice validation failure whenever an order-scoped amount is present.
  It does not capture that Ghostfolio can legitimately return nullable
  `Order.currency`, and it does not distinguish between contradictory source
  data and source fields that are simply absent because the dependency does not
  provide them. That makes the mixed-currency validation rules stricter than the
  empirically supported Ghostfolio contract.

### plan.md

- **Affected sections**: `Technical Context` constraints, `Constitution Check`
  external API review, `Full-History Sync Rules`, and `Verification Additions`.
- **Impact**: The plan currently states that order-currency `unitPrice`,
  `value`, and `fee` stay tied to `currency`, and that incomplete currency
  context is rejected before persistence. That needs to be narrowed so the
   current slice preserves nullable Ghostfolio monetary fields as retrieved,
   rejects only contradictory or impossible currency context that Ghostfolio does
   explicitly provide, and defers later "do not use untraceable financial data"
   rules to future slices. Verification also needs coverage for activities where
   `Order.currency` is `null`.

### contracts/ghostfolio-sync.md

- **Affected sections**: `Successful Response Shape`, `Minimum Required Inputs
  Per Retrieved Activity`, `Normalization And Validation Rules`, and the
  authenticated-user base-currency notes.
- **Impact**: The Ghostfolio contract artifact currently names relevant fields
  but does not specify clearly which response fields are mandatory for this
  slice, which are optional, and which are nullable even when present in the
  payload shape. It needs explicit mandatory-versus-optional field rules for the
  activity contract so validation only requires source data that Ghostfolio is
  contractually expected to provide. That includes documenting nullable
  `Order.currency` semantics and separating "field absent or null but allowed by
  Ghostfolio" from "field missing and required for this slice".

## Root Cause Analysis

BUG-003 correctly identified that monetary values must not be relabeled under
one record-level currency field, but the follow-up artifacts overcorrected by
making per-amount currency completeness a mandatory current-slice validation
gate.

The upstream review confirmed field names such as `currency`, `unitPrice`, and
`fee`, but it did not capture the live nullability semantics of
`Order.currency` or document clearly which Ghostfolio response fields are
mandatory versus optional for this slice. That incomplete dependency
understanding propagated into the `003` specification, contract artifact, plan,
tests, and validation tasks, which now treat a null order currency as invalid
synced data instead of as a nullable upstream field to be stored unchanged.
