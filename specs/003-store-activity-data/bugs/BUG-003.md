# Bug Report: Ghostfolio Activity Mapping Loses Currency Context

**Type**: Spec gap
**Severity**: High
**Feature**: `003-store-activity-data`
**Reported**: 2026-05-17
**Status**: Patched
**Patched**: 2026-05-17

## Description

PR review comment `r3254650682` identified that the current `003` activity
contract and storage model do not preserve enough currency metadata to keep
Ghostfolio financial values semantically consistent with the source data.

The current implementation in `internal/ghostfolio/mapper/activity_mapper.go`
reads `unitPriceInAssetProfileCurrency`, `valueInBaseCurrency`, and
`feeInBaseCurrency`, but stores them in one `ActivityRecord` alongside only one
`base_currency` label. That crosses multiple currencies inside a single
normalized record and loses the original order-currency context needed for
later conversion.

The reviewed Ghostfolio activity payload includes additional fields that are not
currently captured in the `003` artifacts:

- `currency`
- `fee`
- `unitPrice`
- `feeInAssetProfileCurrency`
- `SymbolProfile.currency`

Without those fields, the application cannot safely preserve original-currency
values or prove which currency each stored monetary amount belongs to.

## Root Cause Analysis

The `001` and `003` artifacts preserved the rule "no currency conversion in
this slice" but never made currency identity a first-class requirement for each
stored monetary value. That gap propagated into the `003` Ghostfolio contract
example, the normalized activity data model, and the completed DTO and mapper
tasks.

The implementation then followed those artifacts and mixed
`unitPriceInAssetProfileCurrency`, `valueInBaseCurrency`, and
`feeInBaseCurrency` inside one stored record while keeping only one
`base_currency` string. The result is implementation behavior that is
internally consistent with the incomplete artifacts but not with the actual
Ghostfolio data semantics.
