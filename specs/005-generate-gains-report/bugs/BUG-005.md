# Bug Report: Unlock Flow Treats Rejected Token As New Empty Context

**Type**: Spec conflict
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-22
**Status**: Reported

## Description

When the user enters a Ghostfolio security token that cannot decrypt any selected-server protected snapshot, the application currently opens the normal `Sync and Reports` menu as if this were a valid empty local-user context.

Expected behavior carried forward from the protected-sync slice is stricter:

- if no selected-server snapshot unlocks and Ghostfolio authentication succeeds, treat the token as a new isolated local-user context and allow `Sync Data`
- if no selected-server snapshot unlocks and Ghostfolio rejects the token, stay on the unlock screen, show `access denied`, disable `Unlock`, and leave only `Back` available
- keep the rejected token field value on that failed screen state so the user cannot immediately retry from the same unlock instance
- clear the token field only after the user leaves with `Back` and later returns to the unlock screen
- expose `Sync Data` and `Generate Capital Gains Report` only after either an existing snapshot unlocks or Ghostfolio authentication succeeds for a new context

This is not only a missing error message. The current `005` workflow artifacts collapse `different valid token` and `invalid token` into the same unlocked no-data state, which weakens the authentication boundary and contradicts the earlier isolation rules.

This conflicts with the earlier invalid-token rule and leaves no place for a rejected-token back-only state.
- **data-model.md**: `SyncAndReportsContext` validation rules and the `locked -> unlocked` transition omit a rejected-token branch and do not distinguish snapshot unlock from authenticated new-context activation or a failed-auth lockout state on the unlock screen.
- **Upstream carry-forward dependency**: `specs/003-store-activity-data/spec.md` acceptance scenario 3, `FR-032`, `FR-033`, and `contracts/ghostfolio-sync.md` unlock rules require that snapshot-miss plus Ghostfolio rejection ends with token rejection and no local-data change.

## Root Cause Analysis

This bug exists because the `005` slice refactored the earlier sync flow into a reusable `Sync and Reports` context but did not carry forward the full two-branch unlock rule from `003`.

The current `005` artifacts preserved snapshot discovery, no-data menu behavior, and token reuse after a context exists, but they dropped the required distinction between:

- `no snapshot unlocked + Ghostfolio authentication succeeded` -> valid new local-user context
- `no snapshot unlocked + Ghostfolio authentication rejected` -> access denied on the unlock screen, no context activation, `Unlock` blocked, and token clearing only after leaving and re-entering the screen

The workflow contract then hardened that drift by explicitly allowing context entry on snapshot miss alone, and the verification tasks covered only existing-data and no-data readiness after entry instead of the rejected-token negative path.
