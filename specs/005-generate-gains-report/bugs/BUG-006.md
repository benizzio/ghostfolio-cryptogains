# Bug Report: Report Failures Do Not Produce Source-Faithful Diagnostics

**Type**: Spec gap
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-23
**Status**: Patched
**Patched**: 2026-05-23

## Description

Report generation failures currently only require an actionable non-secret user-visible error and cleanup of partial Markdown output. They do not require a local failure diagnostics report.

Expected behavior:

- every report generation error should be able to generate a failure diagnostics report file
- that diagnostics report should include information about the report generation error
- when the error is tied to a specific activity, the diagnostics report should include the original persisted activity data so the issue can be traced back to the Ghostfolio datasource
- diagnostics activity data should be printed from the original persisted normalized record, not from selected report-calculation inputs or derived report rows
- diagnostics reports should print `null` values explicitly because absent source fields are materially useful for troubleshooting
- the existing sync diagnostics report should also be changed so it prints `null` values instead of omitting them

## Root Cause Analysis

This bug exists because `005` scoped report failure handling to user-visible error recovery and Markdown-output cleanup, while diagnostics behavior remained associated with sync failures from earlier slices.

The report calculation layer can fail on source-data conditions that are only obvious after report-specific validation, such as incomplete monetary contexts, non-positive priced quantities from older compatible snapshots, exact-division failures, missing asset identity, or basis allocation failures. The existing artifacts only require a non-secret reference such as source ID and display label in the UI. That is insufficient for troubleshooting because the user or developer needs the persisted source-shaped record, including which source fields are actually absent.

The existing source-faithful diagnostics slice corrected derived or preferred diagnostic fields, but it did not require explicit `null` field rendering. Omitting `null` values weakens diagnostics for this bug because missing source values are often the reason a report-generation precondition failed.
