# Bug Report: Currency Context Selection Fails Instead Of Skipping Currencyless Tiers

**Type**: Spec gap
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-23

## Description

Production report generation failed for a priced `BUY` activity because the order tier contained order financial values but no `order_currency`.

Observed failure:

```text
activity "ae8c8ca0-0feb-4303-bc66-c9e7296e02e6" order currency context is incomplete; provide gross value and fee from that tier only (asset "BTCUSD", source "ae8c8ca0-0feb-4303-bc66-c9e7296e02e6")
```

The offending persisted record had:

```text
order_currency = null
order_unit_price = 8334.372169710421
order_gross_value = 185.7031469366197
order_fee_amount = 0
asset_profile_currency = USD
asset_profile_unit_price = 8334.372169710421
asset_profile_fee_amount = 0
base_currency = EUR
base_gross_value = 165.7771992703204
base_fee_amount = 0
```

Expected behavior:

- `order -> asset -> base` is priority order for selecting one usable single-activity currency context, not a mandate to fail on the first tier that contains any financial values.
- A tier without its currency code is not a selectable currency context and should be skipped before checking whether that tier can supply or derive the required monetary values.
- The selected tier must be the first tier in priority order that has an explicit currency code and can supply or derive the complete required monetary value set for the activity.
- Once a tier is selected, only that tier's financial information may be used for the calculation input. Values from other tiers must not fill gaps.
- Failure should occur only when no tier with an explicit currency code can supply or derive the activity's complete required monetary value set, or when a selected tier has a currency code but cannot supply or derive required values safely.
- Derivation should be allowed where mathematically exact under `FIN-001`, provided all source values used for the derivation come from the same selected tier.
- Derivation from unit measures to whole measures by multiplication must be preferred over derivation from whole measures to unit measures by division. For example, when quantity and unit price are present in the selected tier and gross value is absent, gross value should be derived as `quantity * unit_price` before considering any division-based fallback. Division-based derivation, such as deriving unit price from `gross_value / quantity`, should be used only when the needed unit measure is absent, the whole measure is present, and the division terminates exactly.
- This preference matters because multiplication of finite exact-decimal source values can produce a finite exact-decimal whole value without introducing the non-terminating-division failure mode. A report should not fail solely because a gross whole value is absent when the selected tier has explicit currency, positive quantity, unit price, and fee that can derive that gross value by same-tier multiplication.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 2 - Generate A Yearly Gains And Losses Markdown Report
- **Affected acceptance scenarios**: Story 2 scenario 11; Story 2 scenario 14
- **Affected edge cases**: `A priced activity has an explicit fee value of 0, which remains valid, while an absent fee value remains incomplete and causes context selection to continue or the report attempt to fail`; `The synced dataset contains mixed currency labels across activities`
- **Affected requirements**: `FR-028`, `FR-030`, `FR-031`, `FR-031a`, `FR-031b`, `FR-031c`, `FR-032`, `FR-033`, `FR-034`, `FR-034a`, `FR-035a`, `FIN-002`, `INT-001b`, `QUAL-001`, `SC-003`
- **Gap identified**: The spec says to evaluate contexts in priority order and select the first context that provides the complete required monetary value set, but it does not explicitly define that a tier without an explicit currency code is not a selectable context and must be skipped. The current `Single-Activity Currency Context Definitions` also say `if the chosen context cannot provide every monetary value needed from that activity, the report-generation attempt fails`, which can be read as failing after tentatively choosing the first tier that contains partial financial data instead of continuing to the next tier. The spec also only names unit-price derivation from gross value and quantity in `FR-031c`; it does not define the symmetric derivation needed for asset-profile tiers that provide currency, unit price, fee, and quantity but omit gross value. It also does not define the precedence rule that same-tier multiplication from unit measure to whole measure must be preferred over same-tier division from whole measure to unit measure.

### plan.md

- **Affected sections**: `Technical Context`, `Constitution Check`, `Reporting Domain Design`, `Verification Plan`, `Complexity Tracking`
- **Impact**: The plan must clarify that single-activity currency context selection first requires an explicit tier currency. Tiers without currency are skipped, not failed, even when they contain unit price, gross value, or fee fields. The report calculation design must validate and derive required values only within the selected tier and must continue through later tiers until it finds a usable complete context or exhausts all tiers. Same-tier multiplication from quantity and unit price to gross value must be preferred before any division-based unit-price derivation path is considered. Verification must include production-shaped activity rows where order financial values exist but `order_currency` is absent and asset or base tier can support the calculation, including rows where that support depends on deriving gross value by multiplication.

## Root Cause Analysis

This bug exists because the artifact set conflates two separate checks: whether a tier is a currency context at all, and whether an eligible context can provide the required financial values. The intended rule is to choose the first usable context in priority order. A tier with no currency code cannot be used as the activity currency context, even if some financial fields are present, so it must not stop evaluation of lower-priority tiers.

The implementation drifted toward treating partial order-tier financial data as a terminal incomplete `order` context. That behavior is stricter than the intended priority semantics and rejects production data that has a valid asset-profile or base currency context. The spec gap allowed this because `FR-031c` only describes one derivation direction and the prose around `chosen context` can be read as failing too early.
