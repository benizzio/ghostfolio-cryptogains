# Bug Report: Sync Data Screen Still Shows Editable Security Token

**Type**: Implementation drift
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-22
**Status**: Reported

## Description

Inside the unlocked `Sync and Reports` context, selecting `Sync Data` still renders the `Ghostfolio Security Token` label and masked token input.

Expected behavior for this slice is different:

- the token is entered once on the unlock screen
- the active context reuses that runtime token for `Sync Data`
- the internal `Sync Data` screen explains that the existing context token will be used and does not ask for, display, or allow editing of the token again

Current implementation evidence shows this is more than a cosmetic issue. The in-context `Sync Data` screen still uses the old token-entry renderer and keeps the token input focusable. `startSync()` reads from that field, so the user can still alter the active context token from the internal sync screen instead of only at unlock time.

## Root Cause Analysis

This bug comes from implementation reuse rather than missing product intent. The unlocked-context `Sync Data` path was built on top of the earlier standalone sync-entry workflow instead of getting a distinct in-context screen and interaction mode.

That reuse carried forward three behaviors that this slice no longer allows:

- the shared sync screen renderer always prints the token label and masked input
- the shared sync flow still exposes token-input focus inside the unlocked context
- the sync start path still reads the token from the visible input field rather than enforcing the already-unlocked context token as the only source of truth

The test suite drifted with the implementation. Contract coverage did not check the `Sync Data Screen` rules from the feature contract, and integration coverage now asserts the wrong behavior instead of catching it.
