# Bug Report: BUG-009 Follow-Up Leaves Proportional Allocation Incomplete And Allocation Rules Inconsistent

**Type**: Implementation drift
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-24
**Status**: Patched
**Patched**: 2026-05-24

## Description

BUG-009 corrected the largest failure mode by separating internal 16-decimal calculation precision from non-rounded Markdown rendering.

Current implementation review still found follow-up gaps:

- fragment-level proportional proceeds allocation required by the report artifacts is not modeled or calculated
- the cost-basis allocation formulas are still inconsistent across artifacts and implementation for rounded partial-fragment allocation
- upstream activity response and synced-history validation still contain exact unit-price division gates that can reject otherwise valid histories before report generation applies BUG-009's rounded internal precision policy
- several supporting artifacts and diagnostics expectations still use superseded `exactly derive` or `same-tier exact derivation` wording

Observed implementation evidence:

- `spec.md` still requires proportional proceeds allocation when one liquidation draws from multiple matched acquisition fragments
- the Go report model has no `BasisMatch` or `matched_proceeds` structure, and the calculator only carries aggregate `AllocatedBasis`, aggregate `NetLiquidationProceeds`, and aggregate `GainOrLoss`
- lot disposal matches are calculated inside lot-based basis state and then discarded before the calculated report model is built
- partial-lot, average-cost, and scope-local fallback allocation currently use direct proportional allocation `(basis * quantity) / total_quantity`, while the active method formulas in `spec.md` still describe rounded unit-cost intermediates before multiplication
- existing BUG-009 tests use repeating-decimal cases such as `1 / 3`, but they do not include a differentiating case such as disposing `2` units from `1` basis over `3` total quantity, where `rounded unit cost then multiply` and `direct proportional division` produce different 16-decimal results
- `internal/ghostfolio/validator/response_validator.go` still calls `decimalsupport.DivideExact(grossValue, quantity)` in `requireDerivableUnitPrice`, so a gross-value-only upstream row whose same-tier unit-price derivation repeats can fail during response validation before mapping or report calculation
- `internal/sync/model/activity_amount_resolution.go` still calls `decimalsupport.DivideExact(*grossValue, record.Quantity)` in `resolveUnitPrice`, so a normalized record whose unit price is derivable only through a repeating division can fail synced-history validation before the protected snapshot is persisted
- `internal/sync/model/activity_amount_resolution_internal_test.go` still asserts that `10 / 3` unit-price derivation fails as `unit price basis input is not exact`, and Ghostfolio response-validator tests only cover exact derivation success rather than repeating-division acceptance

Expected behavior:

- preserve BUG-009's shared 16-decimal internal round-half-up calculation policy for all required division and proportional-allocation sites
- remove or replace pre-report exact unit-price division gates so gross-value-only same-tier derivation that requires repeating-decimal math is not rejected before the report calculation path can apply the shared 16-decimal policy
- when one liquidation spans multiple matched acquisition fragments, calculate and retain fragment-level proportional proceeds and fragment-level gain or loss according to the report artifacts
- reconcile the artifact set so one allocation rule is authoritative for partial-lot, average-cost, and scope-local fallback rounding behavior
- add verification that can distinguish the chosen allocation rule instead of using only cases where both approaches happen to produce the same visible result
- add sync-boundary and report-generation verification for gross-value-only activity rows where `gross_value / quantity` repeats, proving the row remains usable when all other same-tier requirements are met
- remove remaining superseded `exactly derive` or `same-tier exact derivation` wording from supporting artifacts and diagnostics expectations

## Root Cause Analysis

This bug exists because BUG-009 implementation work concentrated on replacing report-blocking exact-division failures with one shared 16-decimal round-half-up division policy at the most visible calculation sites.

That patch corrected report-layer same-tier unit-price derivation and repeating-decimal basis allocation enough for the covered regression fixtures, but it did not carry the same review all the way through pre-report sync validation, fragment-level proportional proceeds allocation, or the remaining artifact wording. At the same time, the verification fixtures chosen for rounded allocation mostly use `1 / 3` shapes, where direct proportional allocation and rounded-unit-cost-then-multiply allocation happen to produce the same visible result. Because those tests do not distinguish the two formulas, and because no sync-boundary fixture proves repeating gross-value-only unit-price derivation remains accepted before report generation, the implementation and artifact conflicts remained hidden while BUG-009 was marked complete.
