# Bug Report: Report Diagnostics Omit Error Cause Chain

**Type**: Spec gap
**Severity**: Medium
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-23
**Status**: Reported

## Description

Current report-failure diagnostics artifacts persist only one `failure_detail` string.

When report calculation, rendering, output preparation, or diagnostics generation wraps a lower-level error, the generated `.diagnostic.json` artifact loses the rest of the causal chain. The artifact can therefore preserve only whichever layer was flattened into `failure_detail` at capture time, which makes production troubleshooting hard because it does not show how the failure moved from the actionable report error into the deeper wrapped cause.

Expected behavior:

- keep the current actionable non-secret `failure_detail`
- add a separate structured field for the ordered non-secret error-cause chain associated with that failure, for example `failure_cause_chain`
- preserve cause order deterministically from the actionable outer failure toward the deepest non-secret wrapped cause
- apply the same redaction and suppression rules used for diagnostics so nested causes do not leak tokens, JWTs, raw payload fragments, or other protected content
- make the ordered cause chain available in generated report-failure diagnostics artifacts so calculation errors are understandable without losing the original offending persisted activity-record context

## Root Cause Analysis

This bug exists because the diagnostics artifact set was defined around one actionable `failure_detail` string and one source-faithful offending-record payload.

The implementation still uses wrapped errors internally for report calculation and runtime orchestration, but the shared diagnostics context and persisted diagnostics document flatten that structure into one string. Because the current specification, plan, and verification tasks only require `failure_detail`, the loss of the remaining wrapped-cause context was never rejected by the artifact set or by tests.
