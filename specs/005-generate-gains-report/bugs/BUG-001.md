# Bug Report: Zero-Priced Holding Reduction Rows Overconstrain Preserved Zero Values

**Type**: Spec gap
**Severity**: Medium
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-21

## Description

`specs/005-generate-gains-report/data-model.md` currently states that, for explained zero-priced holding reductions, `gross_value`, `fee_amount`, and `activity_currency` are blank.

Production verification shows that valid explained zero-priced `SELL` rows can preserve explicit zero-valued reporting inputs rather than only absent values. At minimum, `unit_price`, `gross_value`, and `fee_amount` can be explicit `0` values for the same zero-priced holding-reduction row shape already admitted by `specs/003-store-activity-data/spec.md`.

The current `005` artifact set does not distinguish:

- monetary values not being required for calculation
- preserved zero-valued source fields being forbidden or forced blank in the report model and Markdown contract

That gap matters because explicit `0` is not equivalent to missing data elsewhere in this feature set. It also breaks the carry-forward semantics from `003`, where explained zero-priced `SELL` records are defined by `unit_price = 0` plus an explanatory comment. The current wording pushes the implementation, fixtures, and tests toward dropping valid zero-valued source details for explained zero-priced holding reductions.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 2 - Generate A Yearly Gains And Losses Markdown Report
- **Affected requirements**: `FR-018`, `FR-029`, `FR-031a`, `QUAL-001`, `INT-001`, `INT-001a`
- **Gap identified**: `FR-031a` correctly says an explained zero-priced holding reduction requires no monetary context for calculation, but the `005` artifacts do not separately define how explicit zero-valued preserved fields such as `unit_price`, `gross_value`, and `fee_amount` must be represented when they are present in synced data. That omission allowed later artifacts to tighten the rule into "these fields are blank" instead of "these fields are optional, may be preserved as explicit `0`, and are not required for calculation".

### upstream dependency trace

- **Affected artifact**: `specs/003-store-activity-data/spec.md`
- **Affected requirements**: `FR-017`, `FR-018`
- **Carry-forward gap**: `003` defines a valid explained zero-priced `SELL` as a stored record with `unit_price = 0` and an explanatory comment, preserved for future reporting use. `005` carries forward the non-taxable holding-reduction behavior, but not the possibility that the preserved zero-valued fields themselves can remain present in reporting inputs and detail rows.

### plan.md

- **Affected sections**: `Technical Context` constraints, `Reporting Domain Design`, `Verification Plan`
- **Impact**: The plan needs to distinguish missing monetary context from explicit zero-valued preserved fields for explained zero-priced holding reductions. It should require report-model, rendering, and verification rules that preserve valid zero-valued source details such as `unit_price`, `gross_value`, and `fee_amount` when present without treating them as priced liquidation inputs or required calculation inputs.

### tasks.md

- **Affected tasks**: `T025`, `T029`, `T031`, `T032`, `T033`, `T038`, `T040`
- **False completions**: `T025`, `T029`, `T031`, `T032`, `T033`, `T038`, and `T040` should be reopened because they encode or verify the stricter "blank monetary fields" assumption for explained zero-priced holding reductions and currently drop `unit_price` from that valid stored row shape as well.
- **Missing tasks**: Add a regression task for production-shaped explained zero-priced `SELL` rows with explicit zero `unit_price`, `gross_value`, and `fee_amount`; update report-model and Markdown-contract coverage to distinguish explicit `0` from missing; add implementation coverage that preserves zero-valued detail-row fields without creating proceeds, gains, losses, priced-liquidation behavior, or a false priced-activity currency-context requirement.

### Additional impacted artifacts

- **data-model.md**: `AssetCalculationInput` does not define whether `unit_price = 0` may be preserved for explained zero-priced holding reductions, while `AssetActivityDetailRow` currently says `gross_value`, `fee_amount`, and `activity_currency` are blank.
- **contracts/markdown-report.md**: The in-year activity table rules currently require `Gross Value`, `Fee`, and `Activity Currency` to be left blank for explained zero-priced holding reductions.
- **checklists/calculation.md**: `CHK010` should be reconsidered because the artifact set was marked clear on activity-type-specific monetary requirements, but it did not cover optional preserved zero-valued fields for zero-priced holding reductions.

- **quickstart.md**: The manual scenarios mention explained zero-priced `SELL` holding reductions, but the expected results do not verify whether explicit zero-valued preserved fields remain visible or are intentionally omitted.

## Root Cause Analysis

The `005` report slice correctly carried forward the rule that explained zero-priced `SELL` records produce no gain or loss and do not require a monetary context to calculate basis removal. During report modeling and contract writing, that rule was narrowed too far into a rendering rule that assumes the row has no preserved zero-valued fields at all.

Production verification exposed that the earlier synced-data slice can still preserve explicit zero-valued fields for valid explained zero-priced `SELL` rows, including the defining `unit_price = 0` shape from `003`. Because `005` did not distinguish optional preserved zero values from absent values, the stricter assumption propagated into the data model, Markdown contract, checklist signoff, unit fixtures, and implementation.
