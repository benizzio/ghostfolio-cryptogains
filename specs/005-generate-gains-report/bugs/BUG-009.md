# Bug Report: Exact-Division Rule Blocks Valid Cost Basis Calculations

**Type**: Spec conflict
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-23
**Status**: Patched
**Patched**: 2026-05-23

## Description

The current artifact set extends this slice's no-rounding report-output rule into internal report calculations.

Observed behavior:

- Report calculations require divisions to terminate exactly.
- Non-terminating divisions fail report generation, even for otherwise valid activity histories.
- Complex cost basis methods are affected, especially average-cost updates, proportional proceeds allocation, partial-lot basis allocation, and other per-unit calculations.

Expected behavior:

- Keep the no-rounding rule at the report presentation boundary only.
- Allow rounding for internal report calculations and derivations.
- Use a fixed internal decimal precision of 16 decimal places for report calculations.
- Render report values canonically from those internal calculation results without adding a separate report-boundary rounding step.

## Root Cause Analysis

This bug exists because the current artifacts collapse two different concerns into one rule: report presentation precision and internal calculation precision.

The slice correctly requires canonical non-rounded report rendering, but the current research, plan, requirements, and tests interpret that as meaning all calculations must remain exact and any non-terminating division must fail. Cost basis methods inherently require divisions for average unit cost, proportional proceeds distribution, partial-lot basis allocation, and derived unit values. Some valid histories therefore fail under the current exact-division rule. Because the artifact set never defined a separate internal calculation precision policy, the implementation and tests converged on failure instead of 16-decimal rounded calculations.
