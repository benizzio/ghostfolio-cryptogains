# Bug Report: Asset Identity Key Uses Wrong SymbolProfile Field

**Type**: Dependency issue
**Severity**: High
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-22

## Description

`specs/005-generate-gains-report/spec.md` currently defines the stored Ghostfolio asset identity key as `symbolProfileId` from the upstream activity contract.

The reviewed upstream activity shape actually exposes this identity under `Activity.SymbolProfile.id`. Because the report groups holdings, liquidations, reopening checks, and output sections by this stable key, the wrong field name can cause valid activity payloads to map an empty or incorrect `asset_identity_key`.

Current implementation evidence shows the same assumption has already propagated into DTO and mapper code in `internal/ghostfolio/dto/activity_page_response.go` and `internal/ghostfolio/mapper/activity_mapper.go`.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 2 - Generate A Yearly Gains And Losses Markdown Report
- **Affected requirements**: `FR-012a`, `FR-012b`, `INT-001a`
- **Gap identified**: `FR-012b` names the wrong upstream field and path. The requirement currently says the preserved key comes from `symbolProfileId`, but the activity contract path is `SymbolProfile.id`. That makes the report's asset-grouping requirement depend on an external field that does not match the reviewed Ghostfolio response shape.

### upstream dependency trace

- **Affected artifact**: `specs/003-store-activity-data/contracts/ghostfolio-sync.md`
- **Affected section**: The activity-contract review currently calls out `symbolProfileId` separately from the nested `SymbolProfile` object and does not identify `SymbolProfile.id` as the stable reporting identity source.
- **Carry-forward gap**: `005` explicitly relies on the earlier `003` contract review for asset identity. The incorrect dependency assumption therefore propagated into later feature requirements and implementation planning.

### plan.md

- **Affected sections**: `Technical Context` storage assumptions, `Reporting Domain Design`
- **Impact**: The plan should state that `asset_identity_key` is preserved from `Activity.SymbolProfile.id`, not `symbolProfileId`. Any DTO, mapper, snapshot-versioning, and refresh assumptions tied to the old field name need reevaluation against the actual upstream contract.

### tasks.md

- **Affected tasks**: `T004`, `T005`
- **False completions**: `T004` should be reopened because it explicitly instructs implementation to persist the key from non-empty Ghostfolio `symbolProfileId`.
- **Missing tasks**: Add a regression task to verify `Activity.SymbolProfile.id` maps into `AssetIdentityKey` end-to-end. Add a follow-up task to assess whether snapshots, fixtures, or compatibility checks created under the wrong field assumption need refresh or revised handling.

## Root Cause Analysis

This bug comes from an incorrect dependency assumption rather than from a missing report rule. The feature reused upstream contract knowledge from the earlier sync slice and translated the asset identity source into `symbolProfileId`. That assumption then propagated into `005` requirements, planning, tasks, and code.

Because the identity key is an internal grouping input rather than a user-visible output, the feature artifacts did not include a direct contract check for the exact JSON path. The result is a spec-to-implementation chain that is internally consistent but externally wrong.
