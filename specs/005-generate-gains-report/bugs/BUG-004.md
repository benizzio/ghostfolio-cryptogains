# Bug Report: Disabled Report Action Still Receives Keyboard Focus

**Type**: Spec gap
**Severity**: Low
**Feature**: `005-generate-gains-report`
**Reported**: 2026-05-22
**Status**: Open

## Description

Inside the `Sync and Reports` menu, when `Generate Capital Gains Report` is disabled because synced data is not ready, pressing the arrow keys still moves the selection through that disabled row.

Expected behavior for this keyboard-driven screen is:

- keep the disabled report action visible with its unavailable reason
- prevent the selection cursor from landing on that disabled row
- move arrow-key navigation only between enabled actions such as `Sync Data` and `Back To Main Menu`

Current implementation evidence shows the disabled row still participates in navigation. `internal/tui/flow/navigation.go` updates `m.sync.MenuIndex` by raw index movement, and `tests/integration/sync_reports_context_flow_test.go` currently uses two `KeyDown` presses to reach `Back To Main Menu` when the report action is unavailable.

## Artifact Traceability

- **Affected success criteria**: `SC-001`

Because that rule is missing, the implementation and tests both treat the disabled row as a valid navigation stop.

- **Impact**: The plan already covers visibility and gating for the report action, but it does not state that disabled menu items must be skipped during arrow-key navigation. Verification also needs an explicit regression check that `Sync and Reports` navigation lands only on enabled actions when report generation is blocked.

## Root Cause Analysis

This bug comes from a requirements and coverage gap rather than from a direct contradiction of the current written behavior. The feature artifacts define that the report action stays unavailable until synced data is ready and that the screen is keyboard-driven, but they never state whether a disabled row can still receive selection focus.

The implementation therefore used simple sequential index movement in the unlocked-context menu. The integration test then preserved that assumption by navigating through the disabled row instead of asserting that disabled actions are skipped. The result is a confusing but non-blocking UX bug in the main keyboard flow.
