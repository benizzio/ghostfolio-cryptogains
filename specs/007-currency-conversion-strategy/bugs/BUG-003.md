# Bug Report: Rate Source Summary Repeats Rate-Level Evidence

**Type**: Implementation drift  
**Severity**: Medium  
**Feature**: 007-currency-conversion-strategy  
**Reported**: 2026-06-23

## Description

Production generated Markdown reports can render the `Rate Source Summary` as repeated provider blocks when a report uses one base currency but contains multiple exchange-rate values. The observed output repeats `Authority`, `Provider`, `Rate Kind`, `Unavailable-Date Rule`, `Quote Direction`, and `Rate Value` entries in the summary.

The report has exactly one selected report base currency, and this feature selects one official provider for that base currency. The `Rate Source Summary` should therefore be a concise provider-level summary for the selected base currency. Rate-specific fields such as `Quote Direction` and `Rate Value` are unnecessary in that summary and should remain in the per-activity `Currency Conversion Audit` evidence instead.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 2 - Use Official Historical Conversion Rates.
- **Affected requirements**: FR-020, FR-026, SC-003, and the `Exchange Rate Evidence`, `Conversion Audit Entry`, and `CapitalGainsReport` key entities.
- **Gap identified**: No missing source requirement was found. The specification requires auditable conversion evidence and reproducibility, but rate-specific details belong to converted activity audit evidence. The implementation drift is that the generated `Rate Source Summary` presents per-rate evidence as repeated provider summary content.

### contracts/markdown-report.md

- **Affected sections**: `Rate Source Summary` and `Conversion Audit Section`.
- **Gap identified**: The contract already defines `Rate Source Summary` as concise source metadata for the selected base currency and lists only authority, provider, rate kind, and unavailable-date rule as required content. It does not require `Quote Direction` or `Rate Value` in this summary. Those fields are required in the conversion audit table or equivalent per-activity audit details.

### plan.md

- **Affected sections**: Summary, Project Structure, Conversion Boundary And Rounding, Integration Anticorruption Layer, and Testing Strategy.
- **Impact**: The plan routes canonical rate evidence into report calculation and Markdown audit rendering, but the rendered source summary currently appears to use rate-level evidence directly instead of summarizing the provider selected for the report base currency. The plan and validation work need to distinguish provider-level source summary rendering from per-activity conversion audit rendering.

### tasks.md

- **Affected tasks**: T033, T043, T044, WU11, and WU16.
- **False completions**: T033 is marked complete, but Markdown contract coverage did not prevent repeated source-summary provider blocks or forbid `Quote Direction` and `Rate Value` in the summary. T044 is marked complete, but the rendered summary can repeat for multiple rate values and includes rate-specific fields that the contract reserves for audit evidence.
- **Missing tasks**: Add remediation tasks to update Markdown contract/unit coverage so `Rate Source Summary` renders once per report/provider, contains only provider-level source metadata, excludes `Quote Direction` and `Rate Value`, and keeps rate-specific evidence in `Currency Conversion Audit` or equivalent per-activity audit details.

## Root Cause Analysis

The source summary rendering appears to have been built from the report's rate evidence collection rather than from provider-level metadata for the selected report base currency. Because rate evidence varies by activity date, quote direction, or rate value, a report with several conversions can render multiple provider-summary blocks even though all conversions use the same report base currency provider.

The completed Markdown tests allowed this drift because they asserted presence of provider details but did not assert summary cardinality or field boundaries. At least one renderer-level test expects `Rate Value` in the source summary, which conflicts with the Markdown contract's concise provider-summary intent.

2. Run `/speckit.bugfix.verify` to confirm the spec, contract, plan, and tasks consistently separate provider summary from rate-specific audit evidence.
