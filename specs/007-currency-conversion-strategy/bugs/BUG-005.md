# Bug Report: Currency Conversion Audit Emits Zero Amount Rows And One Row Per Amount Kind

**Type**: Spec gap  
**Severity**: Medium  
**Feature**: 007-currency-conversion-strategy  
**Reported**: 2026-06-24  
**Status**: Patched
**Patched**: 2026-06-24

## Description

Production generated Markdown reports can render several `Currency Conversion Audit` rows for the same activity `Source ID`, one row per amount kind such as `unit_price`, `gross_value`, and `fee_amount`. The observed output also audits explicit zero fee conversions, for example `fee_amount` from `0` to `0` with the same rate evidence used by the activity's non-zero amounts.

This bloats the audit section and makes the report harder to review. Zero-to-zero rows do not add conversion evidence because the converted amount is unchanged regardless of the rate. Repeating one row for every amount kind under the same `Source ID` also hides the activity-level relationship between unit price, gross value, fees, rate date, quote direction, and rate value.

Expected behavior: only real, calculation-relevant conversions should be audited. The rendered audit should use one compact row per converted source activity, or an equivalent compact representation, that shows all relevant converted amount kinds for that source without emitting zero-to-zero audit rows.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 2 - Use Official Historical Conversion Rates.
- **Affected requirements**: User Story 2 acceptance scenario 6, FR-008, FR-020, FR-023, FR-026, SC-003, and the `Converted Activity Amount`, `Conversion Audit Entry`, and `CapitalGainsReport` key entities.
- **Gap identified**: The specification requires audit details for converted priced activities and requires explicit zero values to remain zero, but it does not define audit row cardinality or state that zero-to-zero converted amount slots should be omitted from the rendered audit. Existing wording can be implemented as one rendered row per amount kind, which causes the reported bloat.

### contracts/markdown-report.md

- **Affected sections**: `Conversion Audit Section` and `Monetary Tables`.
- **Gap identified**: The Markdown contract requires an `Amount Kind` column in the compact audit table and states that audit details include amount kind, original amount, and converted amount. It does not require one compact rendered row per source activity, does not define how multiple amount kinds should be grouped in one row, and does not forbid explicit zero-to-zero amount rows.

### data-model.md

- **Affected sections**: `ConvertedActivityAmount`, `ConversionAuditEntry`, and `CapitalGainsReport`.
- **Impact**: `ConversionAuditEntry` already groups multiple `ConvertedActivityAmount` values under one activity through its `amounts` field, but the rendering contract does not preserve that activity-level grouping. The data model also does not distinguish retained zero-valued conversion handling from report-visible audit rows that provide meaningful conversion evidence.

### plan.md

- **Affected sections**: Summary, Conversion Boundary And Rounding, Performance Validation, Integration Anticorruption Layer, and Testing Strategy.
- **Impact**: The plan currently describes compact per-amount audit rows using the BUG-004 column order. It needs to describe grouped per-source audit rendering, zero-to-zero audit row suppression, and tests that prevent audit output from growing by amount-field count when the activity-level rate evidence is shared.

### tasks.md

- **Affected tasks**: T033, T042, T045, T067, T078, T079, WU11, WU15, and WU16.
- **False completions**: T033, T045, T078, and T079 are marked complete, but they did not prevent one rendered audit row per amount kind or zero-to-zero audit rows. T067 is marked complete for zero-valued monetary field conversion behavior, but it does not appear to cover report-visible audit suppression for zero-to-zero conversions. T042 may need reopening if audit entries are recorded or exposed as per-amount rendered rows instead of grouped per-source activity evidence.
- **Missing tasks**: Add remediation tasks to update the Markdown contract, define grouped audit row content, suppress zero-to-zero audit rows, add Markdown contract tests for one rendered row per source activity, add renderer unit tests for grouped amount-kind display, and adjust report model validation if it assumes every converted amount slot must be rendered as its own audit row.

## Root Cause Analysis

The previous audit cleanup narrowed the column set but kept the audit table shaped around amount-level evidence. That preserves reproducibility but does not define a user-facing row cardinality rule. As a result, the renderer can expose the internal `ConvertedActivityAmount` granularity directly and emit separate rows for `unit_price`, `gross_value`, and zero `fee_amount` values that all share the same activity and rate evidence.

This is a spec gap rather than pure implementation drift because the current artifacts do not state that the rendered audit must be grouped by source activity or that zero-to-zero converted amount slots are not report-visible audit rows. The implementation needs a patched contract before changing output shape.
