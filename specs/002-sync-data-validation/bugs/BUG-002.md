# Bug Report: Clipboard Paste Is Missing From Focused Input Flows

**Type**: Spec gap
**Severity**: Medium
**Feature**: `002-sync-data-validation`
**Reported**: 2026-05-10

## Description

The current setup-origin input and Ghostfolio security-token input do not define or verify paste support. The reported expectation is that normal paste actions should work for both fields and should not trigger TUI workflow actions while the field is focused.

This affects these flows:

- entering a custom Ghostfolio server origin during setup
- entering the Ghostfolio security token during `Sync Data`

## Artifact Traceability

### spec.md

- **Affected user story**: Story 1 - Complete Initial Setup
- **Affected user story**: Story 3 - Validate Ghostfolio Communication
- **Affected requirements**: `FR-002`, `FR-003`, `FR-007`
- **Gap identified**: The spec requires focused inputs and suppression of conflicting plain-character hotkeys, but it does not require support for terminal paste sequences or common paste shortcuts while those inputs are focused.

### plan.md

- **Affected sections**: `Full-Screen TUI Rules`; `Constraints`
- **Impact**: The plan needs explicit input-routing rules for paste events and modifier-based paste shortcuts so input handling remains compatible with normal terminal paste behavior.

### tasks.md

- **Affected tasks**: `T013`, `T021`, `T019`
- **False completions**: none; the missing behavior is not currently specified
- **Missing tasks**: add contract, integration, and key-routing coverage for paste handling in setup and sync inputs; implement paste-safe focused-input handling

## Root Cause Analysis

The current artifacts and tests model input entry as single-key typing only. Because paste behavior was not specified, no contract or routing tests cover it and the implementation does not have a defined non-interfering paste path.
