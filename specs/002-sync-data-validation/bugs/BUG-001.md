# Bug Report: Focused Input Traps Setup And Sync Workflows

**Type**: Untested flow
**Severity**: High
**Feature**: `002-sync-data-validation`
**Reported**: 2026-05-10
**Status**: Patched
**Patched**: 2026-05-10

## Description

While the custom-server origin input or Ghostfolio security-token input is focused, pressing `Enter` does not return the user to the primary menu or advance the workflow.

This was reported in both of these flows:

- configuring a new custom server during setup
- entering the Ghostfolio security token during `Sync Data`

## Artifact Traceability

### spec.md

- **Affected user story**: Story 1 - Complete Initial Setup
- **Affected user story**: Story 3 - Validate Ghostfolio Communication
- **Affected requirements**: `FR-002`, `FR-003`, `FR-007`
- **Gap identified**: The spec and workflow contract define focused text inputs and `Up` / `Down` / `Enter` primary-menu navigation, but they do not define the completion path from a focused input back to the actionable menu when the user presses `Enter`.

### plan.md

- **Affected sections**: `Full-Screen TUI Rules`; `Summary`
- **Impact**: The plan needs an explicit rule for what `Enter` does while a labeled input is focused so setup saving and sync validation remain reachable through the advertised primary-action workflow.

### tasks.md

- **Affected tasks**: `T013`, `T021`, `T019`
- **False completions**: `T019` is marked complete even though focused-input exit behavior is not covered
- **Missing tasks**: add focused-input completion coverage for setup and sync workflows; align setup and sync key routing with the documented primary-action path

## Root Cause Analysis

The focused-input branches in the setup and sync flow handlers treat the input as modal and forward `Enter` to the text-input component instead of returning to workflow navigation. Test coverage verifies plain-character typing and menu navigation separately, but it does not verify the transition from text entry back to the primary action path.
