# Bug Report: TUI Header Lacks Persistent Application Identity

**Type**: Spec gap
**Severity**: Low
**Feature**: `002-sync-data-validation`
**Reported**: 2026-05-10

## Description

The current TUI renders generic workflow titles such as `Setup`, `Main Menu`, and `Sync Data`, but it does not include a persistent application identifier. The reported expectation is a small ASCII-art title area with the project name and a Ghostfolio visual cue so the running application is immediately identifiable.

## Artifact Traceability

### spec.md

- **Affected user story**: Story 1 - Complete Initial Setup
- **Affected user story**: Story 2 - Select Sync Data Feature
- **Affected requirements**: `QUAL-002`, `SC-001`
- **Gap identified**: The spec requires a header region and Ghostfolio-inspired visual language, but it does not require a persistent application identity element on startup and workflow screens.

### plan.md

- **Affected sections**: `Full-Screen TUI Rules`; `Project Structure` shared TUI component scope
- **Impact**: The plan needs a shared header or branding component requirement so startup and workflow screens consistently identify the application.

### tasks.md

- **Affected tasks**: `T007`, `T010`, `T017`, `T020`, `T021`
- **False completions**: none; the missing behavior is not currently specified
- **Missing tasks**: add shared branding/header work and screen assertions for startup and workflow screens

## Root Cause Analysis

The design artifacts define screen structure and palette but stop at generic titles. Because no artifact requires a persistent project identifier, the shared layout and screen contracts omit it.
