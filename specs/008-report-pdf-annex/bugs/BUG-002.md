# Bug Report: PDF Renders Literal Markdown Instead Of A Formatted Report

**Type**: Implementation drift
**Severity**: High
**Feature**: 008-report-pdf-annex
**Reported**: 2026-07-05
**Status**: Reported

## Description

The generated PDF appears to contain the correct report data and broad document structure, but it is rendered as literal Markdown syntax inside a PDF instead of as a formatted PDF report. Headings, tables, emphasis markers, and other Markdown source formatting are printed as uninterpreted text, making the PDF difficult for humans to read.

Expected behavior is a human-legible A4 PDF generated from report-domain data through the PDF rendering layer. Markdown-formatted text must not be passed into the PDF renderer as the PDF body. Markdown-specific formatting belongs only in the Markdown renderer and Markdown output files.

The fix should also reassess the PDF rendering research. If the current PDF library can lay out the report with native text, table, heading, spacing, and pagination primitives, the PDF renderer should use those primitives. If it cannot, the implementation must evaluate another local-only option that still satisfies the feature constraints. Any HTML-based option must stay local-only and must not introduce a browser service, remote document service, operating-system print-to-PDF dependency, platform font path dependency, or telemetry path.

## Root Cause Analysis

The existing specification and plan require a dedicated PDF renderer, but the observed output indicates the implementation likely reused Markdown-rendered content or Markdown formatting helpers as the PDF body. This allowed data and rough structure to appear in the PDF while leaking Markdown presentation syntax into the PDF layer.

The completed task state suggests test coverage focused on PDF file creation, text emission, and data presence. It did not fail when the emitted PDF text represented Markdown source instead of formatted report layout. The underlying defect is therefore implementation drift, with a secondary spec/contract hardening need to make the renderer boundary explicit.
