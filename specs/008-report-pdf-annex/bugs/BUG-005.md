# Bug Report: PDF Layout Wastes Width And Clips Split Annex Rows

**Type**: Implementation drift
**Severity**: High
**Feature**: 008-report-pdf-annex
**Reported**: 2026-07-09
**Status**: Patched
**Patched**: 2026-07-09

## Description

After BUG-004 changed every PDF page to landscape, the generated PDF still has production readability defects:

- Tables are unnecessarily narrow and wrap cell text even though substantial horizontal space is unused. Each table must use the available landscape printable width while keeping equal left and right page margins.
- The `Gains-And-Losses Summary`, `Rate Source Summary`, `Reference Section`, `Asset Detail: <asset symbol>`, and `In-Year Activity` subheadings no longer overlap preceding content, but their top margin remains too narrow for readable separation.
- In `Annex 1 - Audit` under `Detailed Per-Asset Audit Report` and `Per-Asset Audit Activity`, a table continuation can cross the bottom printable margin. The page break overlaps and cuts the first continued table line instead of preserving blank bottom margin and complete row information.

## Root Cause Analysis

BUG-004 corrected page orientation and removed overlap, but the renderer and its tests treat fit as a binary no-right-edge-clipping condition. They do not verify that table columns expand to use the landscape printable area or that section spacing remains readable after overlap is eliminated.

The Annex table continuation path also lacks a verified preflight check for the full rendered row height and bottom margin. It can begin drawing a continued row before determining that the row and its borders fit, which conflicts with the existing page-break rule. The primary root cause is implementation drift from FR-005, FR-031, FR-033, and SC-012/SC-013, with specification gaps around balanced width allocation, readable spacing, and bottom-margin preservation.
