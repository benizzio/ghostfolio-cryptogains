# Bug Report: Markdown Initial Detail Labels Are Not Bold

**Type**: Implementation drift
**Severity**: Medium
**Feature**: 008-report-pdf-annex
**Reported**: 2026-07-05

## Description

The generated Markdown main report renders the initial general summary/detail block with plain classifier labels instead of bold labels.

Actual output:

```markdown
- Year: 2025
- Cost Basis Method: Scope-Local Exact Unit Matching, otherwise Scope-Local Average Cost with Oldest-Acquired Deemed-Disposal Order
- Generated At: 2026-07-05 17:36:46 CEST
- Report Calculation Currency: EUR
```

Expected output:

```markdown
- **Year:** 2025
- **Cost Basis Method:** Scope-Local Exact Unit Matching, otherwise Scope-Local Average Cost with Oldest-Acquired Deemed-Disposal Order
- **Generated At:** 2026-07-05 17:36:46 CEST
- **Report Calculation Currency:** EUR
```

The financial values and report structure are not reported as incorrect. The defect is limited to the Markdown presentation contract for the initial report details block.

## Artifact Traceability

### spec.md

- **Affected user story**: User Story 2 - Read A Clearer Main Report (Priority: P2)
- **Affected acceptance scenario**: Scenario 1 states that when the main report has an initial details block with values such as Year and Cost Basis Method, each information classifier label in that block is bold.
- **Affected requirements**: FR-006: "The system MUST render information classifier labels in the initial report details block in bold."
- **Affected success criteria**: SC-002 requires identical PDF and Markdown inputs to contain the same required shared report sections, table columns, text blocks, and values, excluding only pagination, page titles, and Markdown annex file separation.
- **Gap identified**: No missing requirement was found. The specification already requires the expected behavior. The observed Markdown output diverges from the explicit requirement.

### plan.md

- **Affected sections**: Summary; Rendering And Output Boundary; Testing Strategy.
- **Impact**: The plan expects renderer-level presentation changes without changing calculation behavior. The Markdown renderer implementation or its verification must be corrected so the initial details block renders bold classifier labels before the output bundle is considered valid.

### tasks.md

- **Affected tasks**: T024 [US2] Add main report presentation contract tests for bold labels; T029 [US2] Bold initial detail labels and rate-source classifier labels in `internal/report/markdown/renderer.go` and `internal/report/markdown/renderer_conversion.go`.
- **False completions**: T024 and T029 are marked complete, but the reported generated Markdown output still violates FR-006. These tasks need review because either the implementation did not cover the initial details block or the completed tests did not assert the exact Markdown line shape.
- **Missing tasks**: Add or tighten a Markdown rendering regression assertion for the exact initial details format, including `- **Year:**`, `- **Cost Basis Method:**`, `- **Generated At:**`, and `- **Report Calculation Currency:**`; fix the Markdown renderer path that emits the initial details block.

### research.md

- **Affected section**: Main Report Presentation Changes.
- **Impact**: The research classifies bold classifier labels as a presentation rule. The fix should stay in rendering/model presentation code and must not change report calculation inputs or totals.

### data-model.md

- **Affected section**: MainCapitalGainsReport rendering and validation rules.
- **Impact**: The model treats the main report as the primary calculated content rendered before Annex 1. This bug does not require data-model changes unless the bugfix patch decides to make bold initial details an explicit renderer validation rule.

## Root Cause Analysis

The requirement exists in the specification and was planned as a renderer-level presentation change. The completed task state indicates implementation and tests were marked done, but the generated Markdown output still uses plain labels. The likely root cause is implementation drift in the Markdown renderer path for the initial report details block, combined with insufficient regression coverage for the exact Markdown syntax.
