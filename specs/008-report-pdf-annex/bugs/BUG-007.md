# Bug Report: Markdown And PDF Performance Share One Timing Interval

**Type**: Spec gap
**Severity**: Medium
**Feature**: 008-report-pdf-annex
**Reported**: 2026-07-13
**Status**: Reported

## Description

The 10,000-activity performance test applies one two-minute limit to two sequential report-generation runs. It starts one timer before generating Markdown, generates Markdown and PDF in sequence, and checks the combined elapsed time only after both runs complete.

The current CI failure reports `expected SC-007 under 2m0s, got 2m58.843384409s`. This result does not identify the duration of either selected output format and can fail even when Markdown and PDF each complete independently within two minutes.

The inherited performance criterion from feature 005 defines a two-minute limit for one yearly Markdown report run. Feature 008 extends the 10,000-activity scale to PDF but does not define a PDF completion-time criterion or state how separate output-format runs must be measured. The intended correction is to retain the two-minute limit for each format while measuring each `Generate` call independently. Markdown and PDF may execute sequentially in the test, but their elapsed times must never share one timing interval or one aggregate assertion.

## Root Cause Analysis

Feature 005 introduced SC-007 for one Markdown report run because Markdown was the only report output. Feature 008 added a selected PDF output and required both formats to support the same activity scale, but it did not explicitly extend the two-minute completion limit to PDF or define per-format measurement semantics.

During the large-history coverage remediation, the test was expanded from one report output to a loop over Markdown and PDF. The timer remained outside that loop, so two distinct `ReportRequest` and `ReportService.Generate` operations became one aggregate measurement. The assertion continued to cite `SC-007`, although feature 008's SC-007 is unrelated to performance. The artifacts therefore left enough ambiguity for the test to become stricter than the inherited one-run criterion while failing to produce separate Markdown and PDF performance evidence.
